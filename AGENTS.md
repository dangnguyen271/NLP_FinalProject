# AGENTS.md — Codex Instructions for Implementing the NLP Project

## Mission

Implement a complete, tested, reproducible NLP project repository that satisfies the COMP4020 / COMP5040 Natural Language Processing Project proposal brief. The uploaded document is an assignment/proposal specification, not a filled-in project proposal, so do **not** invent a final domain, organisation, dataset, or team roster unless those details already exist in the repository. Build the project so the team can configure those details in one place and still run the full codebase and tests immediately.

The final repository must support:

1. A one-page `proposal.md` generated or maintained from project configuration.
2. A `/data/` directory with a dataset or documented placeholder/sample dataset.
3. `/data/README.md` describing dataset source, provenance, size, structure, domain, suitability, and known challenges.
4. A reproducible NLP pipeline with preprocessing, modelling, evaluation, reporting, and a small demo/prediction interface.
5. Automated tests that pass locally without requiring internet access.
6. A PDF export path for the proposal, with 12 pt Times New Roman or the closest available Times-compatible font and reasonable margins.
7. A checklist reminding the team to keep the GitHub repository private, add instructors as collaborators, include the GitHub repository link in the proposal/PDF, and submit to Canvas.

## Source constraints from the proposal brief

Treat these as non-negotiable assignment requirements:

- The proposal must clearly identify the project title, NLP task, domain/application area, motivation, problem being solved, expected final product, dataset, and team responsibilities.
- The final product can be a sentiment analysis tool, text summarisation system, chatbot/question answering system, topic exploration tool, classification tool, or another NLP system.
- The dataset description must include source/provenance, size/structure, domain, suitability for the NLP task, and challenges such as noisy text, class imbalance, or multilingual data.
- Team roles must cover the NLP pipeline: data collection/preprocessing, feature engineering, modelling/experimentation, evaluation/error analysis, visualisation/reporting, and system/demo development.
- Deliverables include `proposal.md`, dataset in `/data/`, `/data/README.md`, a one-page PDF version of the proposal, and a GitHub repository link in the PDF.
- The repository should be private and must add collaborators `drelhaj` and `whistle-hikhi`.

## Implementation principle

Use a classification-first baseline because it is the safest general implementation for many proposal examples such as sentiment analysis, topic classification, and text classification. Keep the architecture extensible so a later team can swap in summarisation, question answering, topic modelling, or another NLP task.

If project details are missing, create a working default project using a tiny **sample** classification dataset for tests and demos only. Mark sample data clearly as replaceable. Do not present sample data as the team’s final selected dataset.

## Data requirements

Implement a robust data loading and validation layer.

Supported format for the baseline implementation:

- CSV with at least a text column and label column.
- Optional ID column.
- Optional split column with values such as `train`, `validation`, and `test`.

Validation must check:

- File exists.
- Required columns exist.
- Text values are non-empty strings after trimming.
- Label values are non-empty.
- Dataset contains at least two classes for classification.
- Each class has enough rows to avoid obvious split failures; if not, use a deterministic fallback split suitable for tiny test fixtures.
- Duplicated rows are reported.

Create `data/sample_dataset.csv` with a tiny balanced dataset, for example 12 short rows across two labels, so the pipeline can run without a real dataset. Clearly document that this is sample/demo data only.

Create `/data/README.md` with the assignment-required dataset description sections:

```markdown
# Dataset Description

## Source and provenance
## Size and structure
## Domain
## Suitability for the NLP task
## Known challenges
## Replacement instructions
```

## NLP pipeline requirements

Implement these components in `src/nlp_project/`:

### `config.py`

- Load YAML config.
- Provide typed access through dataclasses or Pydantic-style validation without requiring heavyweight services.
- Normalize paths relative to repository root.

### `data.py`

- Load CSV dataset.
- Validate schema and values.
- Produce train/test split using the configured seed.
- Return pandas DataFrames or structured objects.

### `preprocess.py`

- Normalize text deterministically.
- Lowercase by default.
- Strip extra whitespace.
- Remove control characters.
- Keep punctuation unless the config explicitly says to remove it.
- Do not use internet-based downloads.

### `features.py`

- Build a scikit-learn `TfidfVectorizer` with configurable `max_features` and `ngram_range`.
- Include preprocessing in a scikit-learn pipeline where practical.

### `model.py`

- Train a deterministic baseline classifier using `LogisticRegression` or `LinearSVC`.
- Save the fitted pipeline with `joblib` to `artifacts/model.joblib`.
- Load the saved model for inference.
- Provide `predict_texts(texts: list[str]) -> list[str]`.

### `evaluate.py`

- Compute accuracy, macro F1, weighted F1, and per-class precision/recall/F1.
- Save machine-readable metrics to `reports/metrics.json`.
- Save `reports/classification_report.txt`.
- Save `reports/error_analysis.csv` with text, true label, predicted label, and correctness.

### `report.py`

- Provide a small summary function for README/reporting.
- Optionally generate a simple confusion matrix image using matplotlib if installed.

### `proposal.py`

- Generate `proposal.md` from `project_config.yaml`.
- Keep it concise enough for one page when rendered.
- Include all required proposal sections:
  - Project title
  - NLP task and domain/application area
  - Motivation and problem statement
  - Expected final product
  - Dataset source/provenance, size/structure, domain, suitability, and challenges
  - Team responsibilities
  - GitHub repository link
  - Submission checklist
- Include instructor collaborator reminder: `drelhaj` and `whistle-hikhi`.

### `cli.py`

Expose a command-line interface. Use Typer, argparse, or Click. Prefer argparse if avoiding extra dependencies.

Required commands:

```bash
python -m nlp_project.cli validate-data --config config/project_config.yaml
python -m nlp_project.cli train --config config/project_config.yaml
python -m nlp_project.cli evaluate --config config/project_config.yaml
python -m nlp_project.cli predict --config config/project_config.yaml --text "example text"
python -m nlp_project.cli generate-proposal --config config/project_config.yaml
python -m nlp_project.cli run-all --config config/project_config.yaml
```

`run-all` must validate data, train the model, evaluate it, generate proposal outputs, and print the created artifact paths.

## Testing requirements

Create a meaningful test suite. All tests must pass with:

```bash
python -m pytest -q
```

Also support coverage when dependencies are installed:

```bash
python -m pytest --cov=src/nlp_project --cov-report=term-missing
```

Minimum tests:

### `tests/test_config.py`

- Loads example config.
- Verifies required top-level sections exist.
- Verifies placeholder warnings do not crash execution.

### `tests/test_data.py`

- Loads the fixture dataset.
- Validates required columns.
- Raises a clear error when the text column is missing.
- Raises a clear error when only one class is present.
- Produces deterministic splits.

### `tests/test_preprocess.py`

- Trims whitespace.
- Lowercases text.
- Removes control characters.
- Handles empty or `None` values safely according to validation rules.

### `tests/test_model.py`

- Trains on the tiny fixture dataset.
- Saves a model artifact.
- Loads the artifact.
- Predicts labels for at least two example strings.
- Produces metrics with accuracy and macro F1 keys.

### `tests/test_cli.py`

- Runs `validate-data` successfully on fixture config.
- Runs `train` and `evaluate` using a temporary artifact/report directory.
- Runs `predict` and confirms output contains a predicted label.

### `tests/test_proposal.py`

- Generates `proposal.md`.
- Verifies all required assignment sections are present.
- Verifies the GitHub repository link placeholder or configured link appears.
- Verifies collaborator reminders appear.
- Verifies `data/README.md` contains the required dataset description sections.

Test design rules:

- Use `tmp_path` for generated files.
- Do not depend on network access.
- Do not depend on the team’s real dataset.
- Keep tests fast, preferably under a few seconds.
- Ensure randomness is seeded.
